# I18n Design

Internationalization for Uldum: per-player locale resolution, complex-script rendering, and a localized-string API that map authors opt into. This doc settles the architecture before code lands.

Today the engine renders Latin-only ASCII strings, hard-codes English everywhere, and gives map authors no way to localize.

---

## 1. Locale source

**Per-player, client-side, BCP 47.** Each client picks its locale at startup. The server runs scripts; it never needs to know any player's locale.

- Locale codes: short BCP 47 (`en`, `zh-CN`, `zh-TW`, `ja`, `ko`, `ar`, `de`, `fr`).
- Default: `en`.
- Configurable per client via a `locale` field in user settings and `--locale <code>` CLI arg.
- Selection persists across sessions.

BCP 47 is the native format on both Windows (`GetUserDefaultLocaleName`, Vista+) and Android (`Locale.toLanguageTag()`). A small shim normalizes script tags (Windows can emit `zh-Hans-CN`; we ship `zh-CN`) — no normalization library needed.

The engine does not propagate a player's locale over the network. Resolution is the client's job.

## 2. File format

**JSON, flat first-level keys per file. ICU MessageFormat placeholders.**

JSON because it has the widest tooling support (i18next, FormatJS, Crowdin, Lokalise, Phrase all native). ICU MessageFormat for placeholders because it's the cross-language industry standard for substitution, plurals, and selects.

Each strings file mirrors the structure of its paired `types/<entity>.json`: two levels of nesting — entity id → property map. The strings file is a subset, containing only the localizable fields:

```json
// strings/zh-CN/abilities.json
{
  "holy_light": {
    "name":      "圣光术",
    "tooltip":   "为目标治疗 {heal} 生命值，消耗 {mana} 法力。",
    "tooltip.3": "为目标治疗 {heal} 生命值，消耗 {mana} 法力。同时驱散负面效果。"
  },
  "consecration": {
    "name":    "奉献",
    "tooltip": "在 {radius} 码范围内对敌人造成 {damage} 点伤害。"
  }
}
```

The fully-qualified lookup key from the engine's perspective is `ability.holy_light.tooltip` (singular `ability` because keys describe one entity). The resolver:

- Maps the singular key-prefix to the plural filename (`ability` → `abilities.json`).
- Takes the next segment as the entity id (`holy_light`) and walks one level into the JSON.
- Everything after the entity id is a flat key on that level — including any dotted suffixes like `tooltip.3`.

So `ability.holy_light.tooltip.3` resolves to `pack["holy_light"]["tooltip.3"]`, not `pack["holy_light"]["tooltip"]["3"]`. The author writes `"tooltip.3"` as one literal property name (it's valid JSON and clearly identifies the override level).

**Placeholders** — ICU MessageFormat syntax. Supported in v1:

- Named substitution: `{damage}`, `{caster}` — direct insertion of the named arg.
- Plurals: `{count, plural, one {1 item} other {# items}}` — English-style plural rules out of the box; the resolver can be extended for Slavic / Arabic / etc.

Not currently implemented: select (`{gender, select, ...}`), number formatters (`{n, number, currency}`), date formatters. Strings authored with these still parse and round-trip; they just render as the literal arg.

No `_meta` block — filename + path carry the locale identifier. If a locale needs a display name for a picker UI ("English", "中文"), it lives in `engine/strings/locales.json`, not per-pack.

## 3. File locations

Strings files mirror the `types/` folder so every entity type has a paired strings file. Locale directories sit under `strings/`.

```
maps/<map>.uldmap/
  types/
    abilities.json            ← gameplay data + default-language strings
    units.json
    items.json
    destructables.json
    doodads.json
    effects.json              ← no strings file (effects aren't player-named)
  strings/
    en/                       ← OPTIONAL — overrides for default-language
      abilities.json            authoring
      units.json
      items.json
      destructables.json
      doodads.json
      text.json               ← non-entity strings (ui, combat, chat, etc.)
    zh-CN/
      abilities.json          ← entries-only — translate what you want, fall
      units.json                through to types/ for everything else
      ...
      text.json
    ja/
      ...

<project>/strings/            ← game-project Shell strings + locale registry
  locales.json                ← registry: { "en": "English", "zh-CN": "中文", ... }
  en/                           used by the game's settings UI to populate a
    shell.json                  locale picker. Game decides which locales it
  zh-CN/                        actually supports.
    shell.json
  ...                         ← Shell UI keys: menu, lobby, settings, connection
                                errors. Keys are entirely the game's choice — the
                                engine presumes nothing about menu structure.
```

Two logical pools:

- **Map pool** (`<map>/strings/<locale>/`) — all in-game text. Owned entirely by the map author. Loaded fresh per session, dropped at session end.
- **Shell pool** (`<project>/strings/<locale>/shell.json`) — Shell UI strings owned by the game project. Each game ships its own keys + locale list. Engine builds (`uldum_dev`) have no shell strings; the LocaleManager's shell pool stays empty.

**Zero engine-shipped strings.** The engine produces zero strings the player ever sees during a session, and ships no `strings/` content of its own — no shell strings, no locale registry. Asset-load failures, validation errors → developer log, never UI. The locale system is pure infrastructure: game projects own all content.

The two pools never cross-look-up.

## 4. Lookup + resolution

**Client-side at render time.** The server ships `{key, args}`; the client looks up the key against its own locale pack and formats.

```
server: CreateTextTag(L("combat.damage", { damage = 150 }), { pos = {x,y,z} })
   ↓ network ↓
client A (en): "You took 150 damage"
client B (zh-CN): "你受到 150 点伤害"
```

The server has no per-player locale state. The packet carries the short key + args dictionary; the client does the lookup + format.

**Routing**: the lookup helper picks a file based on the first dotted segment of the key. Key prefixes are singular (they name one entity); file names are plural (they hold many).

| Key prefix | Pool / file | Raw fallback? |
|---|---|---|
| `ability.*` | `strings/<locale>/abilities.json` | `types/abilities.json` |
| `unit.*` | `strings/<locale>/units.json` | `types/units.json` |
| `item.*` | `strings/<locale>/items.json` | `types/items.json` |
| `destructable.*` | `strings/<locale>/destructables.json` | `types/destructables.json` |
| `doodad.*` | `strings/<locale>/doodads.json` | `types/doodads.json` |
| Anything else | `strings/<locale>/text.json` | (no raw fallback) |

Shell-pool calls (App-layer code) always route to `engine/strings/<locale>/shell.json` regardless of prefix; there are no entity type files at the engine layer.

**Fallback chain**:

For an entity key like `ability.holy_light.tooltip`:

1. `strings/<active_locale>/abilities.json["holy_light"]["tooltip"]` hit → use.
2. `strings/en/abilities.json["holy_light"]["tooltip"]` hit → use. Warn in dev.
3. **Raw fallback**: the entity registry's `raw_string_field`. The registry owns the schema — a schema-known property like `tooltip` that wasn't authored returns `""`, the same way a missing optional numeric field reads back as its struct default. Unknown fields return nullopt and fall through to step 4.
4. Literal key. Reached for non-entity keys without translations, or unknown entity fields. The key surfaces in the UI so authors spot un-translated / mistyped text immediately.

For a non-entity key like `ui.wave_count`:

1. `strings/<active_locale>/text.json["ui"]["wave_count"]` hit → use.
2. `strings/en/text.json["ui"]["wave_count"]` hit → use. Warn in dev.
3. Literal key string `"ui.wave_count"`.

The "schema default" idea is intentionally at the entity registry, not in locale code — the schema lives with the entity. Callers that distinguish "set to empty" from "absent" (e.g. tooltips skipping the body line) check `result.empty()`.

`text.json` follows the same two-level shape as entity files: top-level segregates by category (`ui`, `combat`, `chat`, …); the next level holds the actual strings. Shell keys follow the same shape against `shell.json` instead of `text.json`.

```json
// strings/zh-CN/text.json
{
  "ui": {
    "wave_count": "第 {current} 波 / 共 {total} 波"
  },
  "combat": {
    "kill_message": "{killer} 击杀了 {victim}",
    "crit":         "暴击！{damage}"
  }
}
```

**Universal walk rule**: every locale file is exactly two levels of JSON nesting. From the lookup key:

- If the first segment is an entity prefix (`ability`/`unit`/`item`/`destructable`/`doodad`), it routes to the matching file (`abilities.json`, etc.) and the *second* segment walks into the JSON. Everything after that is a single flat property name on the inner level.
- Otherwise, the key routes to `text.json` and the *first* segment walks into the JSON (used as a category). Same flat-property rule on the inner level.

A dotted suffix like `tooltip.3` is a literal property name on the inner level, not another walk step — JSON accepts dots in keys.

**Implications**:

- **Maps that don't localize at all** ship zero strings files. Engine reads everything from `types/<entity>.json`. Same workflow as today.
- **Maps localizing to one extra language** ship only the non-default locale folder (e.g. `strings/zh-CN/`). Untranslated keys fall through to `types/<entity>.json` for entity strings (or the key literal for `text.json` keys).
- **Maps wanting a fully-localizable-by-default flow** ship `strings/en/` overrides too, treating `types/` files as a non-locale baseline. Optional — both styles are valid.

Never a hard error in release. Dev's warnings + errors accumulate into a console-accessible list so authors can batch-fix.

## 5. Engine-bound keys + raw fallback

A small set of well-known suffixes the engine looks up directly. The map author writes the default-language strings in `types/<entity>.json`; the engine fills in args from the active level / instance data. Localized overrides live in `strings/<locale>/<entity>.json`.

| Key pattern | Looked up when | Raw field in `types/<entity>.json` | Auto-bound args |
|---|---|---|---|
| `ability.<id>.name` | Ability display name | `abilities.<id>.name` | (none) |
| `ability.<id>.tooltip` | Hovering an ability slot | `abilities.<id>.tooltip` | level def: `damage`, `heal`, `range`, `mana`, `cooldown`, `radius`, `duration`, ... |
| `ability.<id>.tooltip.<N>` | Level-N override | `abilities.<id>.tooltip_<N>` *(if present)* | same |
| `unit.<id>.name` | Unit display name | `units.<id>.display_name` | (none) |
| `unit.<id>.description` | Unit hover / info pane | `units.<id>.description` | unit def: `hp`, `damage`, `speed`, `armor`, ... |
| `item.<id>.name` | Item display name | `items.<id>.name` | (none) |
| `item.<id>.tooltip` | Hovering an inventory slot | `items.<id>.tooltip` | item def + instance: `charges`, `level`, `cooldown`, ... |
| `destructable.<id>.name` | Tree / rock / etc. label | `destructables.<id>.name` | (none) |
| `doodad.<id>.name` | Doodad label (rare in-game) | `doodads.<id>.name` | (none) |

The fields in the right column become the canonical default-language strings. A `types/<entity>.json` that doesn't author a schema field reads back as that field's default — `""` for the optional string fields above, same as reading a missing optional numeric field reads back as its struct default. Callers that care about "absent" vs. "set to empty" check `result.empty()`.

Lookup order for level-suffixed keys: engine tries `ability.holy_light.tooltip.<N>` in the locale file first, then `ability.holy_light.tooltip` in the locale file, then the raw fields in `types/abilities.json` in the same order. Authors write the suffixed key only for genuine per-level wording differences.

Everything else under the same hierarchy is **script-defined** — map's Lua code calls `L(key, args)` and decides which keys exist. Script-defined keys live in `text.json`, not the entity files, and have no raw fallback.

For multi-level abilities specifically: prefer the single-key + data-binding form. Three levels with the same wording but different numbers translate as one string per locale, not three. The suffix override exists for the rare case where level 3 genuinely adds new wording ("...also silences for {silence_dur}s").

## 6. API

### Two types, two contracts

- **Plain Lua string** — used for identifiers, asset paths, developer logs, anything the player never reads.
- **`LocalizedString`** — a small Lua table `{ __loc = true, key, args }` with a `__tostring` metamethod. Produced exclusively by `L(...)`. Carries `{key, args}` until rendered.

The engine enforces this at API boundaries: player-facing text APIs accept *only* a `LocalizedString` (passing a plain string is a type error); identifier/log APIs accept *only* a plain string.

### Constructor

```lua
local text = L("ability.holy_light.tooltip", { heal = 400, mana = 65 })
```

- First arg: dotted lookup key (string).
- Second arg (optional): table of named args for placeholder substitution.

Returns a `LocalizedString` handle.

### Behavior

- **Used locally on the host** — `tostring(text)`, `Log(text)`, string concat (`"x: " .. text`): resolves with the host's active locale at coercion time via `__tostring` / `__concat`.
- **Used on a networked API** — `CreateTextTag(text, ...)`, etc.: engine serializes `{key, args}` into the network packet. Each receiving client resolves with its own locale.

### Why not `Format()`?

Lua already has `string.format("Wave %d/%d", 3, 10)`. We don't wrap it. Map authors who need non-localized formatting use the built-in directly.

### Per-player text variants

For text whose *content* (not just translation) differs per player — "You took 150 damage" for the victim vs "Player A took 150 damage" for everyone else — a separate handle type carries the per-player table. Design TBD; out of scope for this doc.

### API audit — which calls require LocalizedString

**Forced `LocalizedString` (player-facing text)**:

| API | Note |
|---|---|
| `CreateTextTag(text, opts)` | Floating world text |
| `SetTextTagText(handle, text)` | Update existing tag |
| `DisplayMessage(text, opts)` *(future)* | Top-of-screen message |
| `SetLabelText(node_id, text)` | HUD label |
| `SetButtonText(node_id, text)` *(future)* | HUD button label |
| `SetTooltipText(node_id, text)` *(future)* | HUD tooltip |
| `ShowDialog(title, body, ...)` *(future)* | Modal dialog |

**Plain string (identifiers, paths, developer log)**:

| API | Note |
|---|---|
| `Log(text)` | Developer log only — never UI |
| `Print(...)` *(if exposed)* | Developer |
| `SetImageSource(node, path)` | Asset path |
| `PlaySound(path, ...)` / `PlayMusic(path)` | Asset path |
| `PlayEffect(name, ...)` / `CreateEffect(name, ...)` | Effect type name |
| `CreateUnit(type_id, ...)` / `MorphUnit(unit, new_type_id)` | Type identifier |
| `AddAbility(unit, ability_id, ...)` / `RemoveAbility` / `HasAbility` | Ability identifier |
| `CreateItem(type_id, ...)` / `GiveItem` | Type identifier |
| `CreateNode(template_id, ...)` / `ShowNode` / `HideNode` | Template identifier |
| `GetTriggerEvent()` / `GetTriggerAbilityId()` / `GetUnitTypeId(unit)` | Identifier return |
| `SaveData(key, value)` / `LoadData(key)` | Storage key + scalar value |
| `Format(template, ...)` ← use `string.format` | — |

**Edge cases**:

- `GetPlayerName(player)` returns plain string. Player names come from the lobby; they're not localized. To display: `L("welcome", { name = GetPlayerName(p) })`.
- `GetUnitTypeId(unit)` returns plain string identifier (`"footman"`). To get the display name: the engine provides a `GetUnitDisplayName(unit)` (returns `LocalizedString`) that looks up `unit.<id>.name`. Same shape for `GetAbilityDisplayName` and `GetItemDisplayName`.

## 7. Font stack

**System fonts, per-codepoint fallback chain. No engine-shipped fonts.**

`Font::init_from_system` walks a platform-curated list of well-known OS font paths at HUD startup. The first that loads becomes the primary; subsequent successes are added as fallbacks consulted per-codepoint. Modern desktop / mobile OSes ship sufficient fonts to render every common script (Latin, Cyrillic, Greek, Arabic, CJK, emoji) — Chrome, VSCode, and most native apps do the same.

```
get_glyph(cp):
  if primary has cp → rasterize from primary
  else for each fallback (in order):
    if fallback has cp → rasterize from fallback
  else → render missing-glyph box
```

**Per-platform chain**:

| Platform | Primary | CJK fallback | Emoji fallback |
|---|---|---|---|
| Windows | `C:/Windows/Fonts/segoeui.ttf` (Segoe UI) | `msyh.ttc` (Microsoft YaHei) | `seguiemj.ttf` (Segoe UI Emoji) |
| macOS | `/System/Library/Fonts/Helvetica.ttc` | `PingFang.ttc` | `Apple Color Emoji.ttc` |
| Android | `/system/fonts/Roboto-Regular.ttf` | `NotoSansCJK-Regular.ttc` | `NotoColorEmoji.ttf` |
| Linux | DejaVu Sans / Noto Sans (common paths) | Noto Sans CJK (common paths) | Noto Color Emoji (common paths) |

Windows / macOS / Android always succeed — these paths are guaranteed on a clean install. Linux is the wild card: distros vary in installed fonts. If no system font loads, text rendering is disabled and the log clearly says so; a game can layer a fallback in via `Font::load_fallback` / `load_fallback_os_path`.

**Game-supplied overrides** (deferred): a game project will be able to register additional fallbacks (or replace the primary entirely) for brand-consistent visuals across platforms. The `load_fallback` / `load_fallback_os_path` entry points are already in place; the wiring (game.json, project asset packing) lands when needed.

**Complex-script shaping** (Arabic ligatures, Devanagari conjuncts, contextual forms) is **not** in v1. The current per-codepoint mapping renders CJK, Latin, Cyrillic correctly because those scripts don't need shaping. Arabic / Hebrew / Indic scripts render glyphs but without proper ligatures or contextual forms. HarfBuzz integration is deferred until a real use case demands it.

**RTL**: not in v1. Hebrew / Arabic text renders left-to-right; visual ordering would need either HarfBuzz or a basic bidi pass.

**Atlas**: raster MSDF at a fixed `m_em_pixels = 32`. SDF / proper subpixel positioning at varying zoom is deferred.

## 8. v1 scope

- Font fallback chain in `Font` — primary face + N optional fallbacks; per-codepoint lookup walks the chain.
- Locale resolution (config + CLI), pack loaders for shell + map, two-pool lookup with fallback.
- `L(key, args)` helper + `LocalizedString` handle (opaque table, no Lua-side coercion).
- API surface enforces LocalizedString on player-facing calls; plain strings logged + dropped.
- Network: `LocalizedString` ships `{key, args}`; each client resolves with its own active locale.
- Engine-bound key conventions for ability / unit / item / destructable / doodad name + tooltip.
- Raw-fallback to `types/<entity>.json` fields when a locale pack is missing the key.
- Sample packs: `en` and `zh-CN` for `test_map`.

Deferred:

- **HarfBuzz** — only needed for complex-script shaping (Arabic ligatures, Devanagari conjuncts). CJK / Latin / Cyrillic / Greek render correctly without it.
- **RTL bidi** — visual ordering of Hebrew / Arabic runs.
- **Per-locale font stacks** beyond the single fallback chain.
- **SDF text atlas** — raster MSDF at a fixed em-size is acceptable for v1.
- **Engine-shipped fallback fonts** — projects supply their own under `engine/fonts/fallback/`.

## 9. Open notes

- **Locale registry** `engine/strings/locales.json`: shape is `{ "en": "English", "zh-CN": "中文", ... }`. Used by the main-menu picker. Map authors don't touch it.
- **Missing pack for selected locale**: client falls back to `en` for that whole pool (shell or map). Logged.
- **Plural rules**: v1 ships English/CJK rules. Adding Slavic / Arabic / etc. is a resolver extension — strings authored with `=N`/`one`/`few`/`many`/`other` keep working as more rules land.
- **String extraction tooling**: a `scripts/i18n-check`-style tool can scan Lua + JSON for hard-coded display strings and emit a stub locale pack so authors don't start from blank. Doubles as a lint that flags missing engine-bound keys. Not part of the engine; lives in `scripts/`.
